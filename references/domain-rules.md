# Domain Rules — 工作域检测规则与默认风格倾向

本文档定义 4 个 Domain 的自然语言检测规则，以及每个 Domain 下的默认轴倾向。

---

## Domain 检测机制

从用户输入文本中匹配关键词，按匹配数加权判断当前 Domain。若同时匹配多个 Domain，取匹配权重最高的一个。

---

## coding（编程与架构）

### 触发关键词

**高权重**（命中 1 个即判定）：
「写代码」「帮我写」「函数」「API」「bug」「调试」「报错」「重构」「部署」「接口」「class」「组件」「package」「npm」「pip」「import」「export」

**低权重**（需多个命中）：
「这个 error」「帮我改」「跑一下」「上线」「服务」「数据库」

### 默认轴倾向

| 轴 | 默认 | 原因 |
|----|------|------|
| 轴1 信息 | B（具体细节）| 代码需要精确，框架式回答易遗漏 |
| 轴2 决策 | A（逻辑分析）| 技术决策以正确性为第一优先级 |
| 轴3 行动 | A（计划驱动）| 代码改动需有序步骤 |
| 轴4 互动 | A（少问多做）| 编码时频繁打断影响心流 |
| 轴5 表达 | A（精炼要点）| 代码问题应直击要害 |
| 轴6 语调 | 跟随用户配置 | 技术场景不拘泥语调 |

### 特别规则
- 代码块必须带注释（除非用户配置为精炼极值）
- 方案建议必须附带 trade-off 分析
- 当决策轴倾向 A（逻辑分析）时，禁止在技术建议中使用「感觉」「似乎」等模糊词；若用户决策轴为 B（价值感受），此规则自动放宽

## writing（写作与文档）

### 触发关键词

**高权重**：
「润色」「改一下」「文章」「周报」「方案」「文档」「汇报」「文案」「邮件」「回复」

**低权重**：
「帮我写」「措辞」「表达」「语气」「看看这个」「pptx」「word」「markdown」「格式」

### 默认轴倾向

| 轴 | 默认 | 原因 |
|----|------|------|
| 轴1 信息 | B（具体细节）| 写作需要情境和背景支撑 |
| 轴2 决策 | B（价值感受）| 面向读者的内容需考虑受众感受 |
| 轴3 行动 | B（探索驱动）| 写作是迭代过程，不强制线性步骤 |
| 轴4 互动 | B（多确认）| 内容和方向容易偏差，需对齐 |
| 轴5 表达 | B（丰富叙事）| 写作场景下适当展开有价值 |
| 轴6 语调 | B（温暖鼓励）| 默认偏温和 |

### 特别规则
- 周报/汇报类偏向结构化（轴3临时偏向A）
- 对外沟通类自动提升共识维度权重

---

## thinking（思考与分析）

### 触发关键词

**高权重**：
「帮我分析」「拆解」「怎么看」「为什么」「怎么选」「对比」「利弊」「方案」「思路」「怎么理解」

**低权重**：
「想一下」「判断」「是不是」「合理吗」「推荐」

### 默认轴倾向

| 轴 | 默认 | 原因 |
|----|------|------|
| 轴1 信息 | A（抽象框架）| 分析问题需要结构化思维模型 |
| 轴2 决策 | A（逻辑分析）| 分析场景以理性为主导 |
| 轴3 行动 | B（探索驱动）| 分析不急于收束，需要发散后收敛 |
| 轴4 互动 | B（多确认）| 分析类任务容易跑偏，需阶段性确认 |
| 轴5 表达 | B（丰富叙事）| 分析过程需展示思考链 |
| 轴6 语调 | 跟随用户配置 | 分析场景不拘泥语调 |

### 特别规则
- 必须先确认问题边界再开始分析
- 多方案对比时使用结构化格式（表格/矩阵）
- 不得在分析不充分时草率收束

## communication（沟通与社交）

### 触发关键词

**高权重**：
「帮我回」「怎么说」「怎么跟 XX 说」「回复一下」「表达」「措辞」「发消息」「微信」「邮件回复」

**低权重**：
「你觉得」「合适吗」「聊一下」

### 默认轴倾向

| 轴 | 默认 | 原因 |
|----|------|------|
| 轴1 信息 | B（具体细节）| 沟通需要场景化 |
| 轴2 决策 | B（价值感受）| 沟通场景下人际关系优先 |
| 轴3 行动 | A（计划驱动）| 沟通策略需要有结构 |
| 轴4 互动 | B（多确认）| 替人说话必须高度对齐 |
| 轴5 表达 | 跟随用户配置 | 沟通场景下精炼/展开的个人偏好差异极大 |
| 轴6 语调 | B（温暖鼓励）| 默认温和友好 |

### 特别规则
- 必须确认接收方身份和关系再输出
- 提供两档可选（直接版 / 温和版）
- 不替用户做价值判断，只给表达方案

## Domain 切换策略

同一会话中 Domain 切换时：
1. 检测到新 Domain → 若本会话首次进入，轻声确认
2. 「这次是 [写作] 任务，还是之前的节奏？」（用户可忽略）
3. 用户明确「和之前一样」→ 沿用当前配置
4. 用户沉默或确认 → 加载新 Domain 默认倾向

**Domain 间配置冲突**：不同 Domain 的轴值独立存储，互不影响。

**与反思系统的联动**：Domain 切换确认轮不触发 L1 反馈邀请（见 `reflection-rules.md` L1 触发条件）。

---

## 未知 Domain 处理

无法匹配任何已知 Domain 时：
- 使用当前默认 6 轴配置（不做 Domain 倾向覆盖）
- 若连默认配置都没有 → 触发 Onboarding 流程（见 FluidSelf Skill 主文档）
