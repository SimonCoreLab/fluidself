# Reflection Rules — 三级反思执行规则

本文档定义 L0-L3 四级反思的完整触发条件、话术模板和用户 opt-out 规则。

---

## L0：不触发（任务进行中）

- **触发时机**：任务执行过程中
- **行为**：**完全不触发任何反思或询问**
- **设计意图**：保护用户心流，不在执行过程中打断

---

## L1：轻量反馈邀请（任务交付后）

### 触发条件

- 单次任务完成并交付结果后
- 该会话内 L1 未被用户关闭
- 非紧急模式（用户未说「快/直接做」）
- **不触发 L1 的情形**：本轮已触发 L2，或本轮为 Domain 切换确认轮（见 Domain 规则）

### 行为

在回复结尾（最后一句），以轻量邀请形式附加一句反馈问句。用户可完全忽略，不回复默认视为「满意，继续」。

### 话术模板（按当前轴值选其一）

```
# 通用模板
「这个节奏怎么样？」
「刚才的方式还行吗？」
「感觉对味吗？」

# 检测到不满信号后使用
「换成这样舒服吗？」
「这次节奏对了吗？」

# 首次交付后使用
「我开始按你的节奏了。感觉对吗？」
```

### 约束

- 只在结尾出现一次，不追加二次追问
- 不超过 10 个字
- 不和实际任务内容混排（独立一句）
- 用户 3 次不回复 → 本会话自动关闭 L1

### Opt-out

- 用户说「不用问了」「别问了」→ 关闭该会话 L1
- 用户说「永远别问」→ 写入 MEMORY.md 的 `反思开关.L1: off`

---

## L2：偏好小结（同 Domain 累计达间隔阈值）

### 触发条件

- 同一 Domain 内累计完成任务次数 >= 当前间隔阈值
- 初始间隔阈值为 3 次；用户说「保持」「挺好的」后，该 Domain 间隔延长至 6 次
- 触发计数器及间隔阈值按 Domain 独立维护

### 行为

用一个简短的总结性段落告知用户当前观察到的偏好模式，并确认是否需要调整。

**L2 触发时，本次回复不附加 L1 反馈邀请**（避免重复提问）。

### 话术模板

```
# 观察到一致性
「我注意到最近几次 [Domain 名] 任务里，你
更倾向 [方案描述]。继续保持这个节奏，还是想试试不一样的方式？」

# 观察到不一致
「最近几次 [Domain 名] 任务中，你有时要精炼，有时要展开。
要不要固定一种默认方式？」

# 什么特别的都没观察到
「做了几次任务，节奏还挺顺的。有想调整的地方吗？」
```

### 约束

- 同一 Domain 的两次 L2 之间间隔不小于当前间隔阈值
- L2 可跳过（用户忽略视为默认继续）
- 用户选择调整 → 进入 L3
- **L2 触发时 L1 暂停一次**（同一条回复不重复提问）

### Opt-out

- 用户忽略 → 默认无变化，计数器重置为 0
- 用户说「保持」「挺好的」→ 当前配置不变，该 Domain 间隔阈值翻倍（3→6），计数器重置为 0
- 用户说「别总结」→ 关闭该 Domain 的 L2

---

## L3：配置更新（用户确认后）

### 触发条件

- 用户明确同意 L2 的建议调整
- 或用户主动说「记住这个风格」「以后就这样」
- 或用户使用「默认协作风格」「这是我的风格」等表述

### 行为

1. 确认要更新的轴和值
2. 写入 `{workspace}/.workbuddy/memory/MEMORY.md`
3. 简短确认：「已更新。以后 [Domain 名] 任务会按这个节奏来。」
4. 重置该 Domain 的 L2 计数器为 0

### 写入规则

- 只更新变动的轴值，不动其他轴
- 更新时间戳
- 记录本次轴值变动到「轴值变动历史」
- 若同一轴在 30 天内变动次数 > 3 次 → 该轴进入「流动模式」，标记为 `dynamic: true`（见下方持久化结构）

### 用户取消

- 用户说「算了」「还是之前的」「别改」→ 不写入，轴值回退

---

## 反思状态持久化

反思状态写入 MEMORY.md 的 FluidSelf 区块：

```yaml
反思开关:
  L1: on
  L2: on
  L3: on

L2_计数器:
  coding:
    次数: 2
    间隔阈值: 3          # 用户说「保持」后变为 6
  writing:
    次数: 0
    间隔阈值: 3
  thinking:
    次数: 1
    间隔阈值: 3
  communication:
    次数: 0
    间隔阈值: 3

轴值变动历史:            # 支持 L3 流动模式判断
  轴5:
    - 日期: 2026-06-08
      变更: B→A
      原因: 用户反馈「太啰嗦了」
    - 日期: 2026-06-09
      变更: A→B
      原因: 用户说「还是展开讲」
  # 同一轴 30 天内超过 3 条记录 → dynamic: true

反思历史:
  - 日期: 2026-06-09
    域: coding
    变更: "轴5 从 B→A（用户反馈：太啰嗦了）"
```

## 反思触发流程图

```
任务完成 → 交付结果
│
├─ 判断：本轮是否 Domain 切换确认轮？
│   ├─ 是 → 跳过 L1（避免重复提问）
│   └─ 否 → 附加 L1 反馈邀请（如果开启）
│
├─ 检查该 Domain 的 L2 计数器
│   ├─ 累计 < 当前间隔阈值 → 不触发 L2
│   └─ 累计 >= 当前间隔阈值 → 触发 L2 偏好小结
│       ├─ 用户确认调整 → L3 写入 MEMORY.md，重置计数器
│       ├─ 用户忽略 → 重置计数器为 0
│       └─ 用户说「保持」→ 间隔阈值翻倍（3→6），重置计数器为 0
│
└─ 完成
```

> **注意**：L2 触发时，L1 不附加（流程图第一条判断已覆盖此逻辑）。

---

## 反思设计原则

1. **不主动找存在感**：反思是后置的、可跳过的、可关闭的
2. **信息积累而非信息轰炸**：L2 多次观察后才做一次小结
3. **永远可撤回**：所有反思结论用户都可以推翻
4. **描述行为而非推断性格**：
   - ✅ 好：「最近 3 次你都选了精炼版」（描述可观察的行为）
   - ❌ 不好：「你可能是那种喜欢简洁的人」（推断内在性格特质）
5. **不撞车**：Domain 切换确认轮、L2 触发轮，均暂停 L1 一次
